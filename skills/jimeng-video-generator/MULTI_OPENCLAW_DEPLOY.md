# 即梦双机多 OpenClaw 部署说明

这份说明对应当前最新版 skill，目标是让两台不同电脑上的 OpenClaw：

- 读取同一份飞书表格
- 分别登录不同的即梦账号
- 不会对同一条任务重复提单
- 不会去对方账号的资产页下载对方提交的任务

## 当前方案概述

只使用一个飞书字段：

- `执行归属`

程序会在真正开始处理一条 `待处理 / 部分提交` 记录前，先写入：

- `执行归属 = <machineId>`

然后立刻回读确认：

- 如果归属仍然是自己，才继续提单
- 如果归属不是自己，立刻放弃这条

下载阶段同样只处理：

- `执行归属` 属于本机的记录

这样就能保证：

- A 机器不会在 A 账号里重复提交 B 正在处理的任务
- A 机器不会去 A 的资产页认领 B 提交的任务
- B 机器同理

## 两台机器必须满足的前提

### 1. 同一套 skill 代码

两台机器都必须使用同一版 `jimeng-video-generator` skill。

### 2. 两台机器的 machineId 不同

每台机器都必须配置自己的唯一标识。

优先推荐用环境变量：

```bash
export OPENCLAW_MACHINE_ID=mac-a
```

第二台改成：

```bash
export OPENCLAW_MACHINE_ID=mac-b
```

也可以直接写进各自的 `feishu-direct.json`：

```json
{
  "machineId": "mac-a"
}
```

### 3. 飞书表里已有字段 `执行归属`

当前代码会自动补齐这个字段，但建议你先确认表里已经存在。

## 新任务如何运行

### 提单

1. 读取飞书里的 `待处理 / 部分提交`
2. 仅考虑：
   - `执行归属` 为空
   - 或者已经属于本机
3. 写入自己的 `执行归属`
4. 立刻回读确认
5. 提交前再确认一次归属
6. 归属正确才真正提交到即梦

### 下载

资产页下载时，只会处理：

- 飞书记录 `执行归属` 属于本机
- 或本地 trace 中 `execution_machine_id` 属于本机

所以两台机器可以共用一张表，但下载不会串到对方账号。

## 老任务如何处理

历史上已经存在、但 `执行归属` 为空的 `已提交 / 阻塞 / 待回写` 记录，不在这套新保护里。

对于这些老记录，建议：

1. 先人工确认这条到底是哪台机器 / 哪个即梦账号处理的
2. 再把 `执行归属` 补上

如果不能确认，最稳的做法是：

1. 把状态改回 `待处理`
2. 清空 `执行归属`
3. 让新流程重新认领

## 转交任务给另一台机器

如果你想把一条记录从 A 机器转给 B 机器，必须同时做两件事：

1. 把飞书里的 `执行归属` 清空
2. 把状态改回 `待处理`

否则这条记录仍会被视为归属于原机器。

## 运行建议

虽然当前已经有自动归属保护，仍建议：

- 两台机器使用不同的 `machineId`
- 两台机器使用不同的即梦账号
- 首次验收或人工补跑时尽量错峰，避免同时盯同一条新任务

这样可以进一步降低同时抢占“空归属新任务”的概率。

## 最终检查清单

迁移到另一台机器前，逐项确认：

- 两台机器都使用同一版 skill
- `执行归属` 字段存在
- 两台机器 `machineId` 不同
- 两台机器分别登录不同即梦账号
- 两台机器各自的 `runtimeRoot` 独立
- 两台机器定时触发时间已错开

只要这几项成立，当前这版就能支撑双机同表运行。
