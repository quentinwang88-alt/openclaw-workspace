## 当前基线（已上线逻辑）

当前主补货链路的真实逻辑如下：

### 1. 建议采购量先于阈值判断

主链路先计算：

```python
total_available = available + in_transit_qty
days_with_transit = total_available / avg_sales if avg_sales > 0 else 999
target_qty = avg_sales * purchase_cycle_days
suggested_qty = max(0, target_qty - total_available)
```

也就是说，**是否需要补货**，已经优先由“补货周期”和“当前总可用量（现货 + 在途）”共同决定。

### 2. 当前候选池口径

当前主链路的候选池规则是：

```python
if suggested_qty > 0 or purchase_sale_days <= threshold_days:
    # 纳入 restock_list
```

含义：

- 只要 `suggested_qty > 0`，就说明已经低于补货周期，需要实际补货，必须纳入清单。
- `threshold_days` 不再拦截实际补货 SKU，只负责预警分层和展示。
- `purchase_sale_days > threshold_days` 但 `suggested_qty > 0` 的 SKU，当前会进入“提前补货”分组。

### 3. 当前输出口径

- 默认 `only_actionable = true`，即只展示 `suggested_qty > 0` 的实际补货 SKU。
- 系统内部仍保留：
  - `actionable_list`：实际需要补货
  - `warning_only_list`：仅触发预警、当前建议采购量为 0
- 因此，当前系统已经支持“老板视角”和“排查视角”两种输出，只是默认展示前者。

### 4. 当前优先级逻辑仍是旧口径

当前优先级和分组仍然基于 `purchase_sale_days`（仅现货）：

- `0-3 天 = 极高`
- `4-7 天 = 高`
- `8-10 天 = 中`
- `>10 天 = 低`

因此，“优先级误导”问题依然存在，这正是改动 1 要解决的内容。

## 先决事项：统一主链路与旧导出链路

当前系统存在两条补货相关输出链路：

### 主链路

- `skills/restock_skill_adapter.py`
- `skills/export_restock_to_bitable.py`

这条链路已经是**新口径**：

- 用 `suggested_qty > 0` 决定是否实际补货
- `threshold_days` 只负责分层
- 已有“提前补货”分组

### 旧导出链路

- `skills/inventory-alert/alert.py` 中的 `create_feishu_doc()`

这条链路仍然是**旧口径**：

- 固定使用 `采购 5 + 物流 12 + 安全 2`
- 仍按 `suggested > 10` 才入表
- 紧急程度也还是按旧规则输出

### 研发处理建议

在改动 1 / 改动 2 上线前，必须二选一：

1. **同步修改旧导出链路**
   让 `inventory-alert/alert.py#create_feishu_doc()` 和主链路保持同一补货公式、同一优先级口径。

2. **废弃旧导出链路**
   明确只保留 `restock_skill_adapter.py + export_restock_to_bitable.py` 这一条主链路，避免两套口径并存。

建议优先选 **方案 2**，因为维护一套补货口径比长期维护两套更稳。

## 改动 1：优先级判定改用 `days_with_transit`（含在途可售天数）

### 背景与目标

当前优先级只看 `purchase_sale_days`（= 当前库存 / 日均销量），不考虑在途。导致一个“现货 5 天 + 在途 40 天”的 SKU 被标记为“高优先级”，但实际采购量为 0，报告会误导采购同学。

**目标**：优先级反映“真实紧急度”，即算上在途后还能卖几天。

同时保留“现货风险敞口”视图，避免完全丢掉“万一在途出问题，现货撑不了几天”的信息。

### 方案设计

采用**双优先级并存**：

| 字段名 | 计算方式 | 用途 |
|---|---|---|
| `stock_priority` | 基于 `purchase_sale_days`（仅现货） | 风险预警：如果在途出问题，现货能撑几天 |
| `effective_priority` | 基于 `days_with_transit`（含在途） | 排序和执行依据：真实补货紧急度 |
| `risk_flag` | `stock_priority` 高、但 `effective_priority` 低时打标 | 用于标记“依赖在途”的风险项 |

优先级阈值保持不变：

- `0-3 天 = 极高`
- `4-7 天 = 高`
- `8-10 天 = 中`
- `>10 天 = 低`

### 开发任务拆解

#### 任务 1.1：改造优先级函数

定位 `skills/restock_skill_adapter.py` 当前的 `get_priority()`，建议替换为：

```python
def _calc_priority(days: float) -> tuple[str, int]:
    if days <= 3:
        return ("极高", 1)
    elif days <= 7:
        return ("高", 2)
    elif days <= 10:
        return ("中", 3)
    return ("低", 4)


def _enrich_priority(item: dict) -> dict:
    purchase_days = item.get("purchase_sale_days", 0)
    transit_days = item.get("days_with_transit", purchase_days)

    item["stock_priority"] = _calc_priority(purchase_days)
    item["effective_priority"] = _calc_priority(transit_days)

    if item["stock_priority"][1] <= 2 and item["effective_priority"][1] >= 3:
        item["risk_flag"] = "依赖在途"
    else:
        item["risk_flag"] = None

    return item
```

#### 任务 1.2：保持当前候选池口径，不回退到旧阈值拦截

这一点是本期最重要的约束：

**不要**把候选池改成 `days_with_transit <= threshold_days`。

当前已上线主逻辑已经修正为：

```python
if suggested_qty > 0 or purchase_sale_days <= threshold_days:
    ...
```

本次改动应当保持这个口径不变，只新增优先级字段和展示逻辑。

建议新增一个 `risk_watch_list`：

```python
risk_watch_list = [
    item for item in warning_only_list
    if item.get("risk_flag") == "依赖在途"
]
```

这样既不影响“实际补货清单”，又能把“现货风险敞口”单独暴露出来。

#### 任务 1.3：修改排序与分组逻辑

当前排序和分组都基于 `purchase_sale_days`。

本次改动后建议调整为：

- `actionable_list` 默认按 `effective_priority.rank ASC` 排序
- 同优先级内按 `days_with_transit ASC`
- 若仍需第三排序键，可加 `suggested_qty DESC`

报告分组也要同步切换：

- `极高 / 高 / 中`：基于 `effective_priority`
- `提前补货`：定义为 `suggested_qty > 0` 且 `effective_priority = 低`

说明：当前“提前补货”的业务含义是“已低于补货周期，但尚高于预警阈值”，这和 `effective_priority = 低` 的定义兼容。

#### 任务 1.4：修改报告展示与飞书导出

报告中每个 SKU 同时展示：

```text
SKU-A001 | 现货优先级: 高 | 实际优先级: 低 | 风险标记: 依赖在途
SKU-B002 | 现货优先级: 极高 | 实际优先级: 极高 | 风险标记: —
```

飞书表格新增三列：

- `现货优先级`
- `实际优先级`
- `风险标记`

兼容性建议：

- 现有导出字段 `紧急程度` 可先保留一版
- 其值改为 `实际优先级`，避免下游字段名立即变更导致联动报错

#### 任务 1.5：新增配置项（沿用现有 JSON 配置，不引入 YAML）

当前系统没有 `config.yaml` 这一套配置入口。

建议将新配置挂在现有 JSON 配置里，例如：

```json
{
  "restock": {
    "priority_strategy": "both"
  }
}
```

可选值：

- `"effective"`：只看含在途优先级
- `"stock_only"`：回退到旧逻辑
- `"both"`：双字段并存

默认值建议为 `"both"`。

另外，skill 入参可以允许临时覆盖：

```python
generate_restock_report(..., priority_strategy="both")
```

优先级：

`函数参数 > JSON 配置 > 默认值`

### 验收标准

构造以下 4 个测试用例：

| 用例 | 现货可售 | 含在途可售 | 期望 stock_priority | 期望 effective_priority | 期望 risk_flag |
|---|---|---|---|---|---|
| 正常紧急 | 2 天 | 2 天 | 极高 | 极高 | — |
| 依赖在途 | 5 天 | 45 天 | 高 | 低 | 依赖在途 |
| 在途不足 | 8 天 | 9 天 | 中 | 中 | — |
| 完全充足 | 15 天 | 60 天 | 低 | 低 | — |

另外补 2 个链路验收：

1. `only_actionable = true` 时，`依赖在途` 但 `suggested_qty = 0` 的 SKU 不进入主清单，但会进入 `risk_watch_list`
2. 飞书导出里新增字段能够正确落表

### 工作量估算

建议按 **1.5-2 人天** 估算。

原因：

- 主链路逻辑改造不复杂
- 但需要同步调整排序、分组、飞书导出字段
- 还要处理旧导出链路的同步或废弃

## 改动 2：安全天数按 ABC 分层

### 背景与目标

当前补货周期固定为：

`17 天 = 采购 3 + 物流 12 + 安全 2`

对 A 类高销量 SKU 来说，安全天数偏薄，一旦物流延误容易断货；对 C 类长尾 SKU 来说，安全天数又偏厚，容易积压。

**目标**：安全天数根据 SKU 销量分层动态调整，A 类更厚，C 类更薄。

### 方案设计

ABC 分类采用销量累计占比法（帕累托）：

| 分类 | 定义 | 安全天数 |
|---|---|---|
| A 类 | 按日均销量降序排列，累计销量占总销量前 70% | 5 天 |
| B 类 | 累计销量占 70%-90% | 3 天 |
| C 类 | 累计销量占 90%-100%（长尾） | 1 天 |
| NEW | 新品（上架 < 7 天） | 按 B 类处理（3 天） |

补货周期公式改为按 SKU 动态计算：

```text
lead_time_used = purchase_days + logistics_days + safety_days_by_class[sku_class]
```

示例：

- A 类：`3 + 12 + 5 = 20 天`
- B 类：`3 + 12 + 3 = 18 天`
- C 类：`3 + 12 + 1 = 16 天`
- NEW：`3 + 12 + 3 = 18 天`

### 开发任务拆解

#### 任务 2.1：新增 ABC 分类模块

建议新增：

- `skills/restock_abc_classifier.py`

职责：

- 基于全量 SKU 的 `avg_daily_sales` 计算分类
- 输出 `{sku: "A" | "B" | "C" | "NEW"}`

建议修正后的伪代码如下：

```python
def classify_skus_by_abc(
    sku_list: list[dict],
    a_threshold: float = 0.70,
    b_threshold: float = 0.90,
    new_age_days: int = 7,
    unknown_age_default: str = "B",
) -> dict[str, str]:
    result: dict[str, str] = {}

    new_skus = []
    zero_sales_skus = []
    valid_skus = []

    for sku in sku_list:
        code = sku["sku"]
        age_days = sku.get("sku_age_days")
        sales = float(sku.get("avg_daily_sales", 0) or 0)

        if age_days is not None and age_days < new_age_days:
            new_skus.append(sku)
        elif sales <= 0:
            zero_sales_skus.append(sku)
        else:
            valid_skus.append(sku)

    sorted_skus = sorted(valid_skus, key=lambda x: x["avg_daily_sales"], reverse=True)

    if len(sorted_skus) == 1:
        result[sorted_skus[0]["sku"]] = "A"
    elif sorted_skus:
        total_sales = sum(float(s["avg_daily_sales"]) for s in sorted_skus)
        cumulative = 0.0
        for sku in sorted_skus:
            cumulative_before = cumulative / total_sales
            cumulative += float(sku["avg_daily_sales"])
            if cumulative_before < a_threshold:
                result[sku["sku"]] = "A"
            elif cumulative_before < b_threshold:
                result[sku["sku"]] = "B"
            else:
                result[sku["sku"]] = "C"

    for sku in new_skus:
        result[sku["sku"]] = "NEW"

    for sku in zero_sales_skus:
        result.setdefault(sku["sku"], "C")

    for sku in sku_list:
        if sku["sku"] not in result:
            result[sku["sku"]] = unknown_age_default

    return result
```

关键点：

- 单 SKU 时直接归 `A`
- 零销量 SKU 不崩溃，默认 `C`
- 未知年龄 SKU 不强行归 `NEW`，默认走 `B`

#### 任务 2.2：配置新增安全天数映射（沿用 JSON 配置）

建议继续沿用现有 JSON 配置，不引入 `config.yaml`。

例如：

```json
{
  "restock": {
    "purchase_days": 3,
    "logistics_days": 12,
    "safety_days_by_class": {
      "A": 5,
      "B": 3,
      "C": 1,
      "NEW": 3
    },
    "abc_thresholds": {
      "a_cumulative": 0.70,
      "b_cumulative": 0.90
    },
    "new_sku_age_days": 7,
    "unknown_age_default_class": "B"
  }
}
```

如果没有配置，则默认退回现有固定 17 天逻辑。

#### 任务 2.3：改造建议采购量计算逻辑

当前固定逻辑：

```python
target_qty = avg_sales * purchase_cycle_days
suggested_qty = max(0, target_qty - total_available)
```

改成按 SKU 动态：

```python
def calc_suggested_qty(item: dict, abc_map: dict[str, str], config: dict) -> int:
    sku_class = abc_map.get(item["sku"], config["unknown_age_default_class"])
    safety_days = config["safety_days_by_class"][sku_class]
    lead_time_used = config["purchase_days"] + config["logistics_days"] + safety_days

    total_available = item["available"] + item.get("in_transit", 0)
    target_qty = item["avg_daily_sales"] * lead_time_used
    suggested_qty = max(0, int(target_qty - total_available))

    item["sku_class"] = sku_class
    item["safety_days_used"] = safety_days
    item["lead_time_used"] = lead_time_used
    item["suggested_qty"] = suggested_qty
    return suggested_qty
```

#### 任务 2.4：主流程调用顺序调整

原流程近似为：

```text
拉数据 -> 算 suggested_qty -> 按 threshold_days 分层
```

改造后建议为：

```text
拉数据
  -> 基于全量 SKU 计算 ABC 分类
  -> 为每个 SKU 计算 lead_time_used
  -> 计算 suggested_qty
  -> 保持当前候选池口径：
       suggested_qty > 0 or purchase_sale_days <= threshold_days
  -> 拆分 actionable_list / warning_only_list / risk_watch_list
  -> 报告分层
```

注意：

- ABC 分类必须基于**全量 SKU**
- 不允许只用候选池做 ABC，否则累计占比基数错误
- `threshold_days` 继续只负责预警分层，不作为“是否实际补货”的硬门槛

#### 任务 2.5：`NEW` 分类的分期建议

当前系统没有稳定暴露 `sku_age_days`。

因此建议分两步：

##### 方案 A：本期先做 A/B/C，未知年龄统一按 B

- 可立即上线
- 不依赖额外数据源

##### 方案 B：拿到稳定 `sku_age_days` 后启用 NEW

可选数据源：

- BigSeller 直接字段
- 本地销售明细最早日期推算

如果本期拿不到稳定年龄字段，则：

- 保留 `NEW` 相关配置和代码接口
- 运行时统一走 `unknown_age_default_class = "B"`

#### 任务 2.6：报告与飞书导出新增字段

每个 SKU 增加展示：

- `ABC分类`
- `使用的安全天数`
- `使用的补货周期`

飞书导出建议同步增加：

- `ABC分类`
- `安全天数`
- `补货周期`

这样采购同学可以直接看出“为什么这个 SKU 建议量更高 / 更低”。

### 验收标准

#### 分类准确性

- 跑一遍历史数据
- 手工抽检 10 个 SKU 的 ABC 分类
- 与运营认知基本一致

#### 边界用例

| 用例 | 期望 |
|---|---|
| 只有 1 个 SKU | 归为 A |
| 所有 SKU 销量为 0 | 不崩溃，默认归 C |
| 新品（age < 7） | 若年龄字段可用，归 NEW |
| 年龄字段缺失 | 归 `unknown_age_default_class`（默认 B） |

#### 前后对比

跑一份新旧对比报告，重点观察：

- A 类 SKU 建议采购量应普遍提升
- C 类 SKU 建议采购量应普遍下降
- `actionable_skus` 总量不应异常暴涨

### 工作量估算

建议按 **2.5-3.5 人天** 估算。

原因：

- 新增 ABC 模块
- 主流程改造成“按 SKU 动态补货周期”
- 报告与飞书导出字段同步
- `NEW` 分类存在数据依赖
- 旧导出链路仍需同步处理或废弃

## 上线建议

### 1. 分两个 PR 上线

- **PR 1：改动 1（双优先级）**
  - 风险低
  - 收益直观
  - 不改变建议采购量公式

- **PR 2：改动 2（ABC 分层）**
  - 会改变建议采购量
  - 需要运营和采购适应期

### 2. 改动 2 建议灰度

第一周：

```json
{
  "A": 2,
  "B": 2,
  "C": 2,
  "NEW": 2
}
```

这相当于“先只跑 ABC 分类，不改变真实采购量口径”。

第二周再切换到：

```json
{
  "A": 5,
  "B": 3,
  "C": 1,
  "NEW": 3
}
```

### 3. 监控指标

上线后 2 周建议盯以下指标：

- 断货 SKU 数
- 实际补货 SKU 数
- 库存周转天数
- A 类 SKU 服务水平
- C 类 SKU 库存积压占比

如果周转明显变差，大概率是：

- A 类划分过宽
- 或 `unknown_age_default_class = "B"` 仍偏保守

可优先尝试：

- 将 `a_cumulative` 从 `0.70` 调到 `0.60`
- 或将 `A` 类安全天数从 `5` 回调到 `4`
